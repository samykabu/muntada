# Database Migration Guide

## CRITICAL: Never AI-Generate Migrations

Per Constitution X, database migrations MUST NEVER be generated by AI tooling.
Always use the Entity Framework Core CLI.

## Creating a Migration

```bash
# Navigate to the API project
cd backend/src/Muntada.Api

# Create migration (replace <ModuleName> and <MigrationName>)
dotnet ef migrations add <MigrationName> \
  --project ../Modules/<ModuleName>/Infrastructure \
  --startup-project .

# Example for Identity module
dotnet ef migrations add AddUserEntity \
  --project ../Modules/Identity/Infrastructure \
  --startup-project .
```

## Applying Migrations

```bash
# Apply all pending migrations
dotnet ef database update --startup-project backend/src/Muntada.Api

# Apply to a specific migration
dotnet ef database update <MigrationName> --startup-project backend/src/Muntada.Api
```

## Rolling Back

```bash
# Rollback to a specific migration
dotnet ef database update <PreviousMigrationName> --startup-project backend/src/Muntada.Api

# Remove last migration (if not yet applied)
dotnet ef migrations remove --project backend/src/Modules/<ModuleName>/Infrastructure
```

## Best Practices

- Migrations MUST be forward and backward compatible (for blue-green deploys)
- Always review the generated migration code before committing
- Test migration on a copy of production data before deploying
- Each module has its own SQL Server schema (Constitution I)
