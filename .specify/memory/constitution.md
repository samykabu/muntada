# Muntada Constitution

## Core Principles

### I. Modular Monolith Discipline (NON-NEGOTIABLE)
Every business module (Identity, Tenancy, Rooms, Moderation, Chat, Recording, Files, Billing, Reporting, Admin, Public API) owns its own SQL Server schema, service layer, and integration events. Cross-module communication MUST use integration events or explicit service contracts — never direct database joins across schema boundaries. This preserves future service extraction options.

### II. Server-Authoritative State
All room lifecycle, moderation, recording, and consent state MUST be managed by the server. Clients reflect state received from SignalR events and API responses — they never derive authoritative state from local UI logic. State machines defined in the specification are the single source of truth for allowed transitions.

### III. API-First for Multi-Client Readiness
Every business capability MUST be exposed through versioned REST APIs (/api/v1/) before any UI is built. The React SPA is one consumer among future React Native mobile clients. No business logic lives in the frontend. SignalR carries realtime events; LiveKit carries media — both are complementary transport layers, not substitutes for API contracts.

### IV. Test-First for Critical Paths
TDD is mandatory for: state machine transitions, authorization policy evaluation, billing meter logic, consent enforcement, and audit event generation. Red-Green-Refactor strictly enforced for these domains. Integration tests required for: room join flow, recording lifecycle, file scan pipeline, and webhook delivery.

### V. Invite-Only Security Model
Room access is strictly invite-only. Every join-token issuance MUST validate: active invite, tenant policy, room state, and user membership. Guest listeners are listen-only with no chat, file download, or recording capabilities. No silent impersonation — all admin actions require justification and produce audit events.

### VI. Observability from Day One
OpenTelemetry distributed traces for all critical paths (join-token issuance, room lifecycle, recording, file scan, webhook delivery). Structured logging with correlation IDs spanning API, SignalR, worker, and webhook flows. Metrics for: room joins, concurrent participants, websocket connections, artifact scan times, and PAT usage.

### VII. Explicit Over Implicit
Favor explicit state machines over inferred state. Favor integration events over shared mutable state. Favor idempotency keys over retry-and-hope. Favor opaque identifiers over sequential IDs. Every externally visible decision must be traceable through audit events.

### VIII. Clean Code & Documentation (NON-NEGOTIABLE)
All code MUST follow Clean Code Architecture with clear separation of concerns (Domain, Application, Infrastructure, Presentation layers). DRY (Don't Repeat Yourself) and KISS (Keep It Simple, Stupid) principles are mandatory. All C# public types, methods, and properties MUST have XML documentation comments (`/// <summary>`). Code MUST be self-documenting with meaningful names; inline comments are reserved for non-obvious logic only.

### IX. Component Reusability
React UI elements that appear in more than one place MUST be extracted into reusable components under a shared components directory. Shared components MUST have clearly defined props interfaces. Avoid duplicating UI markup across pages — consolidate into composable, well-typed components.

### X. AI-Safe Database Migrations (NON-NEGOTIABLE)
Database migrations MUST NEVER be generated by AI tooling. All migrations MUST be created using the `dotnet ef migrations add <MigrationName>` command executed in the terminal. AI agents may suggest migration names and review generated migration code, but MUST NOT write migration files directly. This prevents schema drift and ensures EF Core's snapshot tracking remains consistent.

### XI. Comprehensive Testing Strategy (NON-NEGOTIABLE)
Every added feature MUST have corresponding unit tests. All unit tests MUST pass before any Git commit. Playwright MUST be used for integration tests and end-to-end (E2E) tests. Tests requiring manual human validation SHOULD be replaced with Playwright automated tests where feasible; if not feasible, they MAY be skipped with explicit justification in the test file. xUnit is the unit testing framework for C#; React Testing Library for frontend unit tests.

### XII. Aspire-First Local Development (NON-NEGOTIABLE)
Every project in this workspace MUST be runnable locally using .NET Aspire 13.2 as the development orchestrator. The Aspire AppHost project MUST declare all service dependencies (SQL Server, Redis, RabbitMQ, MinIO, LiveKit, frontend SPA) so that `dotnet run --project aspire/Muntada.AppHost` provisions the entire local environment in a single command. Docker Compose MAY remain as a fallback but MUST NOT be the primary local development method. Aspire's built-in dashboard provides service discovery, health monitoring, and distributed tracing — replacing the need for separate Jaeger and manual service wiring during development. All new modules MUST register themselves in the Aspire AppHost upon creation.

## Technology Constraints

- **Frontend**: React SPA (TypeScript), Vite build tooling
- **Backend**: ASP.NET Core 8+ modular monolith (C#), Clean Code Architecture
- **Dev Orchestration**: .NET Aspire 13.2 (mandatory for all local development)
- **Media**: LiveKit OSS (self-hosted)
- **Realtime**: SignalR with Redis backplane
- **Database**: SQL Server with per-module schemas
- **Cache**: Redis (session, presence, room state cache)
- **Queue**: RabbitMQ (async workflows: scan, recording, webhooks, reports)
- **Storage**: MinIO (S3-compatible object storage)
- **Auth**: JWT access tokens + refresh tokens, PATs for public API
- **Encryption**: Transport-only in Phase 1, E2EE-ready architecture
- **Deployment**: Self-managed Kubernetes, GitOps pipeline
- **Region**: GCC only, Saudi PDPL alignment required
- **Payments**: HyperPay/Moyasar for GCC local payment methods
- **Testing**: xUnit (C# unit tests), React Testing Library (frontend), Playwright (integration + E2E)
- **Migrations**: Entity Framework Core CLI only (`dotnet ef migrations add`)

## Development Workflow

- Vertical slices: every epic delivers end-to-end through API → SignalR → LiveKit → SQL → Audit
- Each epic must be independently deployable and testable
- Epics respect dependency order: Foundation → Identity → Tenancy → Rooms → Realtime → Moderation → Chat → Recording → Files → Billing → Reporting → Admin → Public API
- AI-assisted development is the primary development method; specs must be detailed enough for AI agents to implement correctly
- All PRs require passing CI (lint, test, build) before merge
- Local development MUST use `dotnet run --project AppHost` (Aspire) as the primary startup method

### Git & PR Discipline

- **Commit per task**: A separate Git commit MUST be created upon completing each task. Do not batch multiple tasks into a single commit.
- **PR per phase**: A Pull Request MUST be created for each implementation phase. PRs MUST include a very descriptive, detailed summary covering all changes, design decisions, and testing performed.
- **Code review before PR**: The code review skill MUST be run before creating any PR. All issues found during code review MUST be fixed before the PR is submitted.
- **All tests pass before commit**: Unit tests MUST pass before any commit is made. Tests requiring manual validation MAY be skipped only if they cannot be replaced by Playwright tests.

### GitHub Issue Tracking

- A GitHub issue MUST be created for each task planned for execution in a phase before implementation begins.
- Issues MUST be updated to closed/completed status once the corresponding task is done.
- A detailed summary of all tasks implemented MUST be included when the phase is completed.

### Specification Workflow

- The default question limitation in Spec Kit MUST be overridden as needed to fully understand the specification or task being built. Thoroughness in understanding requirements takes priority over speed.

## Governance

This constitution supersedes all ad-hoc decisions. Amendments require updating this document and notifying all team members. Any deviation from module boundaries or security model must be documented with justification.
